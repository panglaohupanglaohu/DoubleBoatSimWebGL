# Anti-Idle Protocol（防空转执行协议）

**创建时间**: 2026-03-16 06:54  
**状态**: 生效中  
**适用范围**: 所有冲刺、重构、联调、排障、验收任务

---

## 0. 核心原则

> **没有文件、没有测试、没有运行结果，就等于没做。**

本协议不判断“主观上是否努力”，只判断“是否留下可验证证据”。

---

## 1. 每 30 分钟的硬证据要求

每个 30 分钟时间片结束时，必须至少产出以下 3 类证据中的 1 类：

### A. 文件证据
- 新建文件
- 修改代码文件
- 修改测试文件
- 修改接口/页面/脚本文件

### B. 测试证据
- 单元测试输出
- 集成测试输出
- smoke test 输出
- 失败测试输出（也算证据，但必须带失败原因）

### C. 运行证据
- API 返回结果
- 页面可见变化
- 脚本执行输出
- 服务启动日志
- 调试日志

如果 30 分钟结束时三类证据都没有，状态必须直接写：

> **任务失速**

禁止使用以下表述替代失速：
- 按计划推进中
- 正在分析
- 正在排查
- 正在思考
- 整体正常
- 即将完成

---

## 2. 汇报前强制检查清单

每次状态汇报前，必须先完成以下检查：

- [ ] 最近 30 分钟新增/修改了哪些文件？
- [ ] 最近 30 分钟执行了哪些命令？
- [ ] 命令输出是什么？
- [ ] 最近 30 分钟有没有新的 API / 页面 / 脚本结果？
- [ ] 当前阻塞是什么？

如果前 4 项都为空，则本次汇报只能写：

> **任务失速：本时间片无文件、无测试、无运行结果。**

---

## 3. 20 分钟思考/排障上限

允许思考、读代码、排障，但单次连续时长不得超过 **20 分钟**。

超过 20 分钟后，必须二选一：

1. **拆成更小任务并立即执行**
   - 例如：
     - 先修 import
     - 先补一个最小测试
     - 先打通一个接口
     - 先让页面出现一个可见变化

2. **直接报告阻塞**
   - 说明：
     - 卡在哪里
     - 已尝试什么
     - 下一步准备怎么试

禁止出现连续数小时处于“阅读/思考/整理/犹豫”状态而没有外部产出。

---

## 4. 失速升级机制

### 30 分钟无证据
状态：**任务失速**

动作：
- 明确写出本时间片无硬产出
- 下一时间片只允许做 1 个最小任务

### 60 分钟无证据（连续 2 个时间片）
状态：**轻微失速**

动作：
- 停止扩展目标
- 只保留 1 个 P0
- 下一时间片必须交出至少 1 条硬证据

### 90 分钟无证据
状态：**明显失速**

动作：
- 输出阻塞说明：
  - 当前卡点
  - 已尝试动作
  - 为什么没结果
  - 下一步最小动作
  - 预计多久拿到第一条证据
- 禁止继续使用“推进中”表述

### 120 分钟无证据
状态：**冲刺失败风险高**

动作：
- 暂停所有扩展任务
- 回退到单点突破策略
- 必须优先恢复一个最小闭环：
  - 一个接口可用 / 一个测试通过 / 一个页面变化 / 一个脚本跑通

---

## 5. 单时间片只允许一个主目标

每个 30 分钟时间片只能有 **1 个主目标**，格式如下：

- 修复 `backend.main` 的 import 错误
- 跑通一个 FastAPI 启动
- 实测一个 AI Native API 端点
- 补一个集成测试

禁止一个时间片同时写：
- 完成 API
- 完成前端
- 完成联调
- 完成测试
- 完成文档

目标一旦过大，就容易重新进入空转。

---

## 6. 汇报模板（强制格式）

每次冲刺汇报必须至少包含：

### 1. 文件改动（硬证据）
- 文件路径
- 新建 / 修改

### 2. 测试结果（硬证据）
- 命令
- 通过 / 失败 / 未执行
- 关键输出

### 3. 运行结果（硬证据）
- API / 页面 / 脚本 / 日志结果
- 是否形成可见效果

### 4. 当前阻塞
- 技术阻塞
- 环境阻塞
- 设计阻塞

### 5. 风险判断
- 正常推进 / 轻微失速 / 明显失速

### 6. 下一个 30 分钟目标
- 只能写 1 个主目标

---

## 7. 复读机防护规则

禁止直接复用上一条状态汇报。

每次汇报都必须重新回答：

- **这半小时新增加了什么证据？**

如果答案是“没有”，那就不能复制上一条“正常推进”的口径，必须改为：

> **任务失速**

---

## 8. 验收规则

以下内容才算真实推进：
- 新代码
- 新测试
- 新接口
- 新页面行为
- 新运行结果
- 新验证证据

以下内容不算推进：
- 计划
- 汇报
- 分析
- 催办
- 愿景
- 口头说明
- 无结果排查

---

## 9. 执行口令

以后所有冲刺统一按这句执行：

> **先看证据，再写状态；没有证据，就报失速。**

---

## 10. 当前起效要求

从本文件落地开始：

1. 所有半小时汇报先查证据
2. 所有 60 分钟以上无产出自动升级为失速
3. 所有时间片仅保留 1 个主目标
4. 不再允许用“分析中/推进中”掩盖无结果状态

---

## 11. 最终目的

本协议的目标不是“增加压力”，而是防止以下情况再次发生：

- 冲刺时长很长，但有效交付极少
- 中间长时间空档却仍维持“正常推进”叙事
- 定时任务变成机械复读而不是状态审计

**以后判断工作是否推进，只看证据，不看感觉。**
