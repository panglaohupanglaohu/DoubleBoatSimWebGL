# AI Native CPS 4 小时改写方案

## 目标

在 4 小时内，把当前项目从“多模块联调样机”提升到“具备主动认知闭环的 CPS 雏形”，优先补齐最影响系统完整性的模块，而不是追求大而全。

## 总体策略

### 改写原则

- 先补控制中枢，再补边缘细节。
- 先补结构化输出，再补展示层消费。
- 先保证可解释、可追溯、可测试，再谈大规模扩展。

### 本轮不做的事

- 不做大范围前端重写。
- 不做真实外部 API 深接入。
- 不做 Hadoop / Kudu / Impala 级重型湖仓部署。
- 不做强化学习控制器和真实 NMPC 求解器接入。

## 4 小时拆解

### 第 1 小时：统一认知语义

目标：让导航、合规、决策说同一种语言。

执行项：

- 为导航模块补 COLREGs 遭遇分类。
- 为合规专家补认知快照中的规则、证据、建议和工程参数区块。
- 修正风险等级合并逻辑，避免字符串比较导致的错误升级。

产出：

- 规则化导航报告
- 更强的认知快照结构

### 第 2 小时：决策任务化

目标：让系统从“报告生成器”变成“任务编排器”。

执行项：

- 决策包增加 `action_plan`。
- 增加 `mission_brief`，面向驾驶台与总师。
- 给每个动作补领域、优先级、执行理由、约束来源。

产出：

- 结构化任务清单
- 可直接被 UI 和自动化层消费的决策包

### 第 3 小时：记忆层与外部态势补齐

目标：让 CPS 具备基本可追溯能力。

执行项：

- 修复 DataLakehouse 查询不包含内存缓冲的问题。
- 增加记忆概况接口数据结构。
- 将 WorldMonitor 占位数据升级为稳定的真实结构 mock。

产出：

- 可回放的记忆层
- 可联调的外部态势输入

### 第 4 小时：文档、验证、交付

目标：让团队能继续接手，不把知识留在代码里。

执行项：

- 重写 README。
- 补充本计划文档。
- 运行核心测试并记录结果。
- 检查 Git 远端与推送可行性。

产出：

- 新 README
- 本轮改写说明
- 验证结论

## 优先级

### P0

- 智能导航 COLREGs 语义化
- 合规专家认知快照增强
- 决策编排器任务化输出
- 湖仓查询一致性修复

### P1

- WorldMonitor 稳定 mock
- 前端聚合器接入任务摘要
- 新文档

### P2

- 云同步分页与列表能力
- 更细粒度测试与性能剖析
- 真正的 RL / NMPC / 多智能体协作控制器

## 成功标准

- 后端能返回结构化 `mission_brief` 和 `action_plan`。
- 导航风险不再只是数字，而是有规则、角色、动作。
- 记忆层查询能看到缓冲中的最新事件。
- 前端聚合器能同时拿到本地 dashboard、协调状态、任务摘要和外部海事态势。
- README 足够让新研发成员在 10 分钟内理解系统结构和启动方式。
